# 上半部分：红石链路

## 1. 范围与目标

通过统一红石控制模式、统一比较器状态信号、GUI 按钮和公开 API，使四台机器接入红石自动化系统（Pipez、vanilla 漏斗+比较器、Redstone 系统）。

### 1.1 范围
- 3 RS 模式（IGNORE / HIGH / LOW）
- 统一比较器三段语义（0 / 1-14 / 15）
- 4 机器 GUI RS 按钮
- 公开 API（`RedstoneControlMode` + `MachineControlInfo` + `setMachineRedstoneMode`）

### 1.2 研究依据
- 业界 RS 模式基线：Thermal Expansion 3 模式（Disabled/Low/High）、Mekanism 同、Industrial Foregoing 3 模式、Pipez 3 模式（含 Always Off）。本模组四机器均为批次加工机器（100–600 tick/批），与 Thermal/Mekanism/IF 类别一致，采用 3 模式与业界批次机器主流对齐。
- Modular Routers 5 模式（含 Pulse）有 Pulse 先例，但它是路由器类（每 tick 循环执行模块），与本项目批次机器类别不同；PULSE 的真实价值集中在灌装机"按钮触发一批"场景，对冷凝器/培养槽/拼接机边际。PULSE 推迟到 v1.2.0 候选，待 v1.1.7 玩家反馈验证真实需求后再决定（见 §16.3）。
- Pipez 1.20.1 红石三模式与本项目 IGNORE/HIGH/LOW 完全对齐（业界验证）。
- 比较器三段语义（0/1-14/15）：vanilla 熔炉容器填充度被玩家普遍理解为"烧炼进度"（混合语义先例）；Mekanism 机器、Productive Bees hive 均明确覆盖默认容器填充度语义；vanilla composter/cake/cauldron/jukebox/lectern/beehive 6+ 种自定义比较器输出。三段语义完全符合 vanilla 与主流模组自定义比较器输出的传统。

## 2. 红石控制模型

### 2.1 模式枚举
新增公开枚举 `RedstoneControlMode { IGNORE, HIGH, LOW }`，使用稳定字符串序列化 `ignore/high/low`。枚举值末尾追加（不替换 ordinal）以便未来 v1.2.0 候选 PULSE 模式扩展。

### 2.2 持久化字段
每个机器方块实体只持久化以下字段：
- `RedstoneMode`（当前模式）

**不持久化**（可从当前世界和机器状态重新计算，写入 NBT 会产生陈旧状态）：
- `RedstonePowered`（当前是否被红石供电）—— 加载时通过 `resamplePowered()` 重新采样
- `lastComparatorSignal`（比较器缓存）
- `processingAllowed`（推导值）

**旧存档兼容**：缺失字段默认 `IGNORE`，不崩溃。NBT 未知字符串 → IGNORE + 一次 warn（M5）。

> **v1.2.0 设计采纳（§3.2）**：原 v1.1.7 计划持久化 `RedstonePowered`，采纳 v1.2.0 §3.2 的更稳健设计——只持久化 `RedstoneMode`，避免加载陈旧供电状态。

### 2.3 信号采样
- `MachineBlock.neighborChanged()` 在服务端读取 `level.hasNeighborSignal(pos)` 并通知方块实体。
- 区块加载时重新采样当前信号。

### 2.4 模式语义
- **IGNORE**：忽略输入，默认模式。
- **HIGH**：有红石信号时运行。
- **LOW**：无红石信号时运行。
- HIGH/LOW 在加工中失去许可时冻结进度，恢复许可后继续，不清零或消耗输入。

### 2.5 红石作用范围
- **冷凝器**：控制产水，手动暂停仍是独立的第二道门；自动向下输送保持独立。
- **培养槽**：控制生长与成熟；资源注入、瓶子输出和自然衰减继续运行。
- **拼接机**：控制拼接开始与进度。
- **灌装机**：控制配方选取、开始与进度。
- 机器不直接输出普通弱/强红石，只提供比较器输出，避免输入与输出形成自锁反馈。

## 3. 统一比较器链路

### 3.1 三段语义（优先级 15 > 1..14 > 0）
- `15`：主产物槽存在可抽取物品。
- `1..14`：`processing=true` 或 `progress>0`，计算 `ceil(progress × 14 / maxProgress)`，clamp 至 `1..14`。
- `0`：无主产物 且 `processing=false` 且 `progress=0`。

### 3.2 主产物定义
| 机器 | 主产物 |
|---|---|
| 冷凝器 | 纯净水瓶 |
| 培养槽 | 成熟培养产物（玻璃瓶副产物不触发 15） |
| 拼接机 | 子代种子 |
| 灌装机 | 灌装成品 |

### 3.3 边界场景决议
- **拼接机两亲本就位但 progress=0**：`splicing=true` 即加工中，输出 `1`，避免启动瞬间 0→1 抖动。公式：`processing ? max(1, ceil(progress×14/max)) : 0`。
- **灌装机 progress=99% 且输出槽已有产物**：15 优先（产物就绪是更紧迫的"该取货了"信号）。判定顺序：`if (mainOutput non-empty) 15 else if (processing || progress>0) 1..14 else 0`。

### 3.4 缓存与更新
- 每个方块实体缓存 `lastComparatorSignal`。
- 状态同步时仅在数值变化后调用 `updateNeighbourForOutputSignal`，避免每 tick 更新比较器网络。

### 3.5 兼容性变更
- 培养槽原有"营养度/7"输出改为统一状态信号。
- 拼接机补齐比较器支持。
- 在升级说明中明确标记为兼容性变更。

## 5. 公开 API

### 5.1 新增类型（不修改现有 DTO 构造签名）
```java
public enum RedstoneControlMode {
    IGNORE, HIGH, LOW
}

public record MachineControlInfo(
    RedstoneControlMode mode,
    boolean powered,
    boolean processingAllowed,
    int comparatorSignal
) {}
```

**API 演进策略（v1.2.0 §10 采纳）**：`MachineControlInfo` record 发布后**不得直接增加组件字段**。未来 v1.2.0 若决定加 PULSE 模式，必须新建 `MachineControlInfoV2` 或新增独立查询 API，不修改现有 record 构造签名。`RedstoneControlMode` 枚举可在末尾追加 `PULSE`（字符串持久化已覆盖旧存档兼容），下游 mod 重新编译适配枚举追加。

### 5.2 CyberCultivatorAPI 新增方法
```java
@Nullable
public static MachineControlInfo getMachineControlInfo(Level level, BlockPos pos);

public static boolean setMachineRedstoneMode(
    Level level,
    BlockPos pos,
    RedstoneControlMode mode
);
```

### 5.3 语义约束
- `getMachineControlInfo`：保持现有 API 的 null-safe 约定，位置无效返回 `null`。
- `setMachineRedstoneMode`：
  - 只允许逻辑服务端执行，客户端调用返回 `false`。
  - **非主线程调用返回 `false` + log debug**（M7 细化）。
  - 参数或目标无效返回 `false`。
  - 走与 GUI 相同的服务端验证、持久化、同步和比较器通知路径。

**线程检查实现（v1.2.0 §10 采纳）**：使用 `MinecraftServer.isSameThread()` 替代 `Thread.currentThread() != getRunningThread()`，更 idiomatic 且语义清晰：
```java
if (!(level instanceof ServerLevel serverLevel)) return false;
if (!serverLevel.getServer().isSameThread()) {
    LOGGER.debug("setMachineRedstoneMode rejected: non-main-thread call at {}", pos);
    return false;
}
```

### 5.4 兼容性
原有 `IncubatorInfo`、`BottlerInfo`、`CondenserInfo`、`SplicerInfo` 保持二进制和源码兼容。

## 9. 通用工业自动化

### 9.1 分面能力矩阵

|---|---|---|---|
| 冷凝器 | 无能力槽 | 纯净水只出 | 纯净水只出 |

### 9.2 无方向查询
暴露全部槽位，但仍执行相同的槽位级输入/输出限制。

### 9.3 WorldlyContainer 改造
- **拼接机**：`Container` → `WorldlyContainer`（原版漏斗兼容）。
- **培养槽**：开放顶部种子自动输入，实现完整无人值守链路。

### 9.4 能力插入规范化
- 沿用机器标准逻辑：种子数量限制为 1、缺失基因 NBT 时补齐默认数据、灌装输入变更时正确取消旧配方进度。
- **模拟插入/抽取**：不得修改库存、进度、配方缓存、红石授权或触发同步。
- **模拟路径与实际加工路径共享 `RecipeOrdering` 选择**（研究补强）。

### 9.5 槽位限制
- 输出槽拒绝外部插入。
- 输入槽拒绝外部抽取。
- 所有操作完整保留物品 NBT。

### 9.6 LazyOptional 生命周期（S2）
- 按 face→角色映射缓存（非 7 个独立实例）。
- `invalidateCaps()` 失效；`reviveCaps()` 重建；始终回退 `super.getCapability()`。

### 9.7 防御性只读
能力读取返回的物品视图不得允许外部直接修改内部 `ItemStack`；必要时在 Forge 分面包装器外增加防御性只读包装。

### 9.8 线程安全策略（v1.2.0 §17 采纳）

**不在生产 capability 中静默拒绝非主线程调用**。

不在每个 `IItemHandler` 方法（`extractItem`/`insertItem`/`getSlots`/`getStackInSlot`/`getSlotLimit`/`isItemValid`）起始处检查调用线程并返回 `EMPTY`。

理由：
- 这不是完整线程安全（BlockEntity 字段仍非线程安全）；
- 会造成自动化模组偶发传输失败（Create/Mekanism/AE2 等可能在非主线程查询能力）；
- 难以诊断——返回 `EMPTY` 看起来像"槽位空"，调用方无法区分"线程被拒"和"实际无物品"。

正确处理方式：
- capability 方法保持同步、短小、无异步任务；
- 所有修改仍通过方块实体统一入口（共享 `MachineInventoryPolicy` 谓词）；
- 开发环境可添加 debug 断言或日志（非生产拒绝）；
- 公共 API 写入（`setMachineRedstoneMode`）继续保持服务端主线程限制（§5.3）。

> **设计变更说明**：原 v1.1.7 §9.8 计划"非主线程返回 `EMPTY`"，采纳 v1.2.0 §17 的更稳健设计——不在 capability 层做线程拒绝。这与 `setMachineRedstoneMode` API 的线程拒绝不冲突：API 是写入入口，capability 是自动化查询入口，二者职责不同。

### 9.9 WorldlyContainer 与 IItemHandler 一致性（S3）
共享槽位级 `canInsert`/`canExtract` 谓词，同一规范化逻辑。

### 16.3 v1.2.0 候选：PULSE 模式

**状态**：v1.1.7 不实施，待玩家反馈验证真实需求后决定是否纳入 v1.2.0。

**业界先例**：
- Modular Routers 5 模式（Always on / High Signal / Low Signal / Always off / Pulse）—— 路由器类，每 tick 循环执行模块，PULSE 语义"信号到达时执行一轮模块扫描后停止"
- Bibliocraft Clock solid/pulse 切换、More Red Pulse Gate —— 红石元件类

**推迟理由（从本模组最优角度）**：
1. 本模组四机器玩法本质是"输入驱动"而非"时间驱动"，PULSE 真实价值集中在灌装机"按钮触发一批"场景，对冷凝器/培养槽/拼接机边际。
2. §2.5 五条脉冲消耗规则是 v1.1.7 上半部分最复杂的逻辑，复杂度收益比偏低。
3. 可逆性更高：v1.1.7 先发布 3 模式最小可行版本；v1.2.0 末尾追加 `PULSE` 不破坏旧存档（字符串持久化已覆盖）；删 PULSE 再加回的破坏性远低于保留 PULSE 再删。
4. 批次加工主流机器（Thermal / Mekanism / IF / Pipez）确实不提供 Pulse，与本项目四机器类别一致。

**v1.2.0 启用条件**（满足任一即可考虑）：
- 玩家反馈明确表达"按钮/计时器触发一批血清"的需求
- 整合包作者反馈需要"多机器同步批次"的红石同步语义
- 内部测试发现 PULSE 在自动化工厂中有不可替代的价值

**v1.2.0 实施草案**（若决定启用）：
- `RedstoneControlMode` 末尾追加 `PULSE`（字符串 `pulse`，ordinal 不替换）
- `MachineControlInfo` 末尾追加 `pulseArmed` 字段（boolean）
- 4 机器持久化新增 `PulseArmed` 字段
- `ContainerData` 同步新增 `pulseArmed`
- 恢复 §2.5 脉冲授权消耗规则（M6）：成功完成/事件取消 = 消耗授权；输入变化/配方重载/输出阻塞 = 不消耗
- GUI 按钮循环改为 4 项 `IGNORE → HIGH → LOW → PULSE → IGNORE`
- i18n 新增 2 条 key（`RS:脉冲` 按钮标签 + tooltip）
- GameTest 覆盖脉冲先到输入后到、输出阻塞、持续高电平、第二次上升沿、存档重载、事件取消消耗授权
- 下游 mod 重新编译适配 API 字段追加（MC mod 生态可接受）
